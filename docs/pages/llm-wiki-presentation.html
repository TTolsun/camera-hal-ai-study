AI Study · 프로젝트 맥락 인프라

LLM Wiki

Wiki는 지식을 빠르게 만들고 연결하는 방식이다.

LLM Wiki는 그 지식을
AI Agent가 실무 작업에 활용하게 만드는
프로젝트 맥락 인프라다.

01 — Wiki란 무엇인가

Wikipedia가 아니라,
협업 방식이다

Wiki의 본질은 문서 저장이 아니라 지식 업데이트 방식이다. 여러 사람이 지식을 다루는 협업 시스템.

  • 01빠르게 만든다 — 진입 장벽 없이 즉시 작성
  • 02계속 고친다 — 공동 편집으로 진화
  • 03서로 연결한다 — 링크로 문서를 잇는다

정적인 문서함이 아니라, 지식이 계속 자라고 연결되는 구조다.

02 — Wiki의 핵심은 "연결"

문서가 연결되면
지식 구조가 된다

폴더에 쌓이면 자료 모음, 서로 연결되면 지식 구조. AI 입장에서 이 연결이 결정적이다 — 어떤 모듈이 어떤 flow에 속하고, 어떤 변경이 어떤 dependency를 건드리는지 따라갈 수 있다.

03 — LLM Wiki란 무엇인가
AI Agent가 프로젝트를 이해하고
작업하기 위한
구조화된 프로젝트 지식 문서

사람용 Wiki에서 AI용 작업 맥락으로 확장한 것.

한 줄 정의 — LLM Wiki는 AI에게 주는 README가 아니라, AI가 프로젝트에서 일할 때 참고하는 작업용 맥락 지도다.

보통 repo 안의 markdown 문서로 두고, Agent가 작업 전에 읽거나 RAG/Search로 참조하게 만든다. Confluence, README, Vector DB 중 어디에 두느냐보다 핵심은 AI가 참고할 맥락을 구조화하고 최신으로 유지하는 방식이다.

05 — 적용 전

코드만 보는 AI는
"좋은 코드"를 말한다

AI의 조언
  • 큰 클래스니까 분리하세요
  • 공통 로직으로 빼세요
  • enum 이름을 단순화하세요
  • 테스트를 추가하세요
하지만 모르는 것
  • HAL lifecycle 제약
  • vendor interface 제약
  • metadata compatibility
  • buffer / fence ownership
  • performance-sensitive path

틀린 말은 아니다. 그러나 실무에서는 위험한 답이 될 수 있다.

06 — 적용 후

맥락을 보는 AI는
"이 프로젝트에서 안전한 변경"을 말한다

  • public / vendor interface는 유지 — 깨면 안 되는 경계 인지
  • capture request path에서는 allocation 금지 — 성능 path 인지
  • metadata 변환은 compat layer에서 처리 — 호환성 인지
  • buffer / fence ownership 변경 금지 — 소유권 규칙 인지
  • helper 분리는 내부 함수부터 시작 — 안전한 리팩터 순서

Before의 AI는 좋은 코드를 말하고, After의 AI는 이 프로젝트에서 안전한 변경을 말한다.

07 — Graph로 보는 LLM Wiki

Graph는 수학이 아니라
관계 지도다

Node = 지식 단위 (module · class · 문서 · rule · issue)

Edge = 관계 (참조 · 의존 · 호출 · 영향 · 제약)

잘 만든 LLM Wiki는 단순 문서 모음이 아니라 프로젝트 지식 그래프가 된다.

10 — 장단점과 운영 원칙

오래된 Wiki는 AI에게
잘못된 확신을 준다

장점
  • AI 답변 품질 안정화
  • 맥락 설명 반복 감소
  • 온보딩 비용 감소
  • 리뷰 기준 공유
  • Agent 성공률 증가
  • 암묵지 보존
단점
  • 초기 구축 비용
  • 방치하면 썩는다
  • 코드와 어긋나면 위험
  • 너무 길면 안 본다
  • 죽은 문서함이 될 위험
운영 원칙
  • 코드 변경 시 관련 Wiki도 같이 수정
  • 큰 설계 변경은 Decision 문서로
  • Agent가 자주 틀린 부분을 Wiki에 반영
  • 작업 단위로 짧게 쪼갠다
  • Owner를 정하고 리뷰 기준에 포함
11 — 결론

LLM Wiki는 예쁜 문서가 아니다.

AI가 잘못 판단하지 않도록 프로젝트의 판단 기준을 외부화하는 작업이다.

프롬프트를 잘 쓰는 팀보다, AI가 참고할 맥락을 잘 관리하는 팀이 AI를 더 잘 쓰게 된다.
